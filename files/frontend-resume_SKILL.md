# 前端简历提炼 Skill

---

## 💡 使用提示（必读）

> **代码片段选择技巧**：本 skill 在分析**指定代码片段**时最准确。以下是最佳实践：

### ✅ 推荐提供的代码类型
- **核心业务组件**：体现复杂交互逻辑的组件（如表单、列表、权限控制）
- **工具函数/composable**：封装了复用逻辑的代码（如请求封装、状态管理）
- **配置文件**：构建配置、路由配置、状态管理配置等
- **有技术亮点的代码**：包含性能优化、设计模式、工程化实践的部分

### ❌ 不推荐
- 纯 UI 展示的简单组件（展示型代码亮点有限）
- 大量重复的模板代码（选择 1-2 个代表即可）
- 不完整的代码片段（至少要能看懂功能意图）

### 📦 建议数量
提供 **3-5 个核心代码片段**即可，不需要上传整个项目。优先选择你**自己写的**、**觉得有技术含量**的部分。

## ⚡ 执行流程总览

> **⛔ 强制要求**：所有步骤均为必做项，严格按序执行，不可跳步、不可合并、不可省略。每完成一步在输出中标注 `[Phase X · Step Y 完成]`，未标注视为未完成。

```
Phase 1 · 信息采集（Step 1-3）
  └─ Step 1 · 用户画像采集       ← ⛔ 必须最先执行
  └─ Step 2 · AI 使用经历采集    ← ⛔ 无论用户是否提及，主动询问
  └─ Step 3 · 转型意向采集       ← ⛔ 无论用户是否提及，主动询问

Phase 2 · 代码分析（Step 4-6）
  └─ Step 4 · 技术栈识别
  └─ Step 5 · 八维度深度分析
  └─ Step 6 · 隐性亮点挖掘       ← 包含 A-F 六类 + 主动追问

Phase 3 · 输出生成（Step 7-10）
  └─ Step 7 · 简历描述生成       ← 含 AI 模块 + 转型模块
  └─ Step 8 · 面试题生成
  └─ Step 9 · 回答模板生成
  └─ Step 10 · 最终汇总输出
```

**执行规则**：
- ✅ 严格按顺序执行，Phase 1 全部完成后才进入 Phase 2
- ✅ 用户未提供的信息：主动询问 → 若仍无法提供 → 根据代码推断并标注 `（推断）`
- ✅ 每步完成后标注进度，便于用户追踪和继续

# Phase 1 · 信息采集

## Step 1 · 用户画像采集

⛔ **本步骤为首要必做项，代码分析前必须完成。**

### 1.1 基础背景（请逐一提供）

请提供以下信息，格式自由，描述清楚即可：

| 信息项 | 提问方式 |
|--------|----------|
| **项目类型** | "你的项目是面向 B 端客户、C 端用户、内部工具，还是开放平台？" |
| **你的角色** | "在这个项目里你是独立开发、核心开发，还是参与开发？" |
| **用户规模** | "产品大概有多少用户？日活或并发量级大概是多少？" |
| **项目周期** | "这个项目做了多久？是短期项目还是长期迭代？" |
| **目标岗位** | "你求职的目标岗位是什么？初级前端/中级前端/高级前端/架构师？" |

### 1.2 工作年限（⭐ 关键决策项）

⛔ **必须明确告知你的工作年限**，不同阶段的简历策略完全不同。

请直接回复，例如："我工作 2 年" 或 "我是应届生" 或 "工作 5 年了"

#### 参考标准（帮助判断）：
- **1-3 年**：初中级前端，定位是"能独立完成任务，有工程意识"
- **3-5 年**：高级前端，定位是"能独立负责模块，有技术判断力，能带动团队"
- **5 年以上**：资深/专家级，定位是"能定义技术方向，降低团队整体复杂度，产生业务价值"

**④ GitHub 上有没有和 AI 相关的项目？**
> 如：独立做过 AI 工具/应用、开源项目有 Star/Fork、写过 AI 相关技术博客

> 以上任一项回答"有"即会在简历中生成 AI 相关描述。如都没有，回答"无"即可。

## Step 5 · 八维度深度分析

逐一扫描，识别技术亮点与设计决策。**每个维度只输出存在的亮点，无亮点则跳过。**

### 5.1 组件设计
- 组件拆分粒度与职责划分（是否单一职责）
- 通信模式：props/emit / provide-inject / 状态管理 / 事件总线
- 复用策略：slot / HOC / render props / composables / hooks
- 设计模式：策略/观察者/工厂/代理/组合模式
- 组件文档与 TypeScript 类型定义质量

### 5.2 请求与数据层
- 请求封装：axios 实例 / 拦截器 / 统一错误处理
- 接口管理：模块化 / 自动生成（OpenAPI 等）/ 版本管理
- 数据缓存：SWR / stale-while-revalidate / 本地缓存
- 请求优化：取消重复请求 / 并发控制 / 请求队列
- 实时通信：WebSocket / SSE

### 5.3 状态管理
- 方案选型及理由
- 全局 vs 局部状态划分策略
- 持久化方案：localStorage / sessionStorage / pinia-plugin
- 异步状态模式：loading / error / data 三态处理

### 5.4 CSS 与样式工程
- CSS 方案及隔离策略
- 主题系统：CSS 变量 / Design Token / 暗色模式
- 响应式：媒体查询 / container query / 自适应布局

### 5.5 技术亮点（按场景识别）

| 场景 | 识别关键词 |
|------|-----------|
| 文件上传 | 分片/断点续传/拖拽/进度 |
| 权限系统 | 路由守卫/按钮权限/动态菜单/RBAC |
| 复杂交互 | 拖拽排序/在线编辑/富文本/可视化配置 |
| 数据可视化 | ECharts/D3/AntV/大屏适配 |
| 国际化 | i18n/语言包/动态切换 |
| 微前端 | qiankun/Module Federation/子应用通信 |
| 低代码/可视化搭建 | schema 渲染/拖拽布局 |

### 5.6 性能优化
- 加载优化：懒加载 / 预加载 / 骨架屏 / SSR / CDN
- 渲染优化：虚拟列表 / memo / shouldUpdate / 防抖节流
- 缓存策略：keep-alive / HTTP 缓存 / Service Worker
- 构建优化：代码分割 / Tree Shaking / 压缩 / 分包策略
- 网络优化：图片懒加载 / WebP / 资源预加载 / DNS 预解析

### 5.7 工程化与质量
- 构建配置深度（Vite/Webpack 自定义配置、插件开发）
- 代码规范：ESLint / Prettier / Husky / commitlint
- 测试覆盖：单元/组件/E2E 及测试策略
- CI/CD：自动化构建 / 部署 / 灰度发布
- Monorepo 管理（Turborepo / Nx 等）

### 5.8 错误处理与监控
- 错误边界（Error Boundary / errorCaptured）
- 全局异常捕获：拦截器 / unhandledrejection
- 日志与监控：Sentry / 自建上报
- 降级策略：接口超时 / 服务不可用兜底方案

## Step 6 · 隐性亮点挖掘

> **💡 给用户的提示**：不用纠结描述是否准确、是否值得写。你只需要简单描述，我会根据你的描述去对应代码中综合分析，挖掘真正的亮点。

### 6.1 请依次描述以下内容（全部必填）

⛔ **以下问题必须全部询问，用户必须逐一回答。**
⛔ **不要接受"没有"作为答案，引导用户深入回忆细节。**

---

**问题 1：封装与复用**
你有没有封装过什么工具函数、composable、组件或 hook？

有没有实现过比较复杂的交互逻辑？

**问题 4：踩坑与解决**

**问题 6：动态配置能力**
产品或运营有没有提过类似这样的话？
> - "这个页面不同角色要显示不同内容"
> - "这个能不能不改代码就能改"
> - "以后运营要自己配置 XX"

## Step 9 · 回答模板生成

每个 P0/P1 面试题提供结构化回答模板：

```
【结论先行】
一句话给出核心答案（不要先铺垫背景）

【技术原理/方案选型】
- 核心机制是什么
- 为什么选这个方案（对比了哪些备选）
- 关键实现细节

【项目中的实际应用】
- 具体场景：（描述业务背景）
- 技术实现：（描述你的方案）
- 遇到的挑战：（描述难点）
- 解决方式：（描述解法）

【结果与收益】
- 量化：（性能/效率/稳定性指标）
- 定性：（如无量化数据，描述业务价值）

【延伸/反思】（可选，用于加分）
- 方案的局限性是什么
- 如果重来，会有哪些改进
```

**完整示例（虚拟列表）**：

```
【结论先行】
用虚拟列表解决了万级数据表格的渲染卡顿问题，首屏渲染从 3.2s 降至 0.8s。

【技术原理】
虚拟列表只渲染可视区域内的 DOM 节点，通过动态计算 scrollTop
偏移量来决定渲染哪些条目。对比了 vue-virtual-scroller 和自研方案，
最终选择自研，因为需要支持动态行高，第三方库支持不够好。

【项目实际应用】
场景：审批记录列表，单次查询返回 5000+ 条数据，产品不允许分页。
实现：用 IntersectionObserver + 高度预估 + 动态修正三步走。
挑战：动态行高导致滚动位置跳动。
解法：用 ResizeObserver 实时修正已渲染行的高度缓存。

【结果】
首屏渲染 3.2s → 0.8s，内存占用减少约 70%，用户投诉归零。

【反思】
若重来，会优先考虑后端分页 + 无限滚动方案，前端虚拟列表维护成本偏高。
```
