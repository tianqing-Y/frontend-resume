# 前端简历提炼 Skill

## 概述

本 Skill 帮助前端开发者从项目代码中提炼简历亮点，生成符合 STAR 法则的简历描述，并生成分层面试题和结构化回答模板，全面支撑求职准备。

---

## ⚡ 执行流程总览

> **⛔ 强制要求**：所有步骤均为必做项，严格按序执行，不可跳步、不可合并、不可省略。每完成一步在输出中标注 `[Phase X · Step Y 完成]`，未标注视为未完成。

```
Phase 1 · 信息采集（Step 1-3）
  └─ Step 1 · 用户画像采集       ← ⛔ 必须最先执行
  └─ Step 2 · AI 使用经历采集    ← ⛔ 无论用户是否提及，主动询问
  └─ Step 3 · 转型意向采集       ← ⛔ 无论用户是否提及，主动询问

Phase 2 · 代码分析（Step 4-6）
  └─ Step 4 · 技术栈识别
  └─ Step 5 · 八维度深度分析
  └─ Step 6 · 隐性亮点挖掘       ← 包含 A-F 六类 + 主动追问

Phase 3 · 输出生成（Step 7-10）
  └─ Step 7 · 简历描述生成       ← 含 AI 模块 + 转型模块
  └─ Step 8 · 面试题生成
  └─ Step 9 · 回答模板生成
  └─ Step 10 · 最终汇总输出
```

**执行规则**：
- ✅ 严格按顺序执行，Phase 1 全部完成后才进入 Phase 2
- ✅ 用户未提供的信息：主动询问 → 若仍无法提供 → 根据代码推断并标注 `（推断）`
- ✅ 每步完成后标注进度，便于用户追踪和继续

# Phase 1 · 信息采集

## Step 1 · 用户画像采集

⛔ **本步骤为首要必做项，代码分析前必须完成。**

### 1.1 基础背景

如用户已提供则直接使用，否则主动询问：

| 信息项 | 说明 |
|--------|------|
| 项目类型 | B端 / C端 / 内部工具 / 开放平台 |
| 角色 | 独立开发 / 核心开发 / 参与开发 |
| 用户规模 | 日活 / 并发量级（影响性能亮点权重） |
| 项目周期 | 几个月 / 迭代期 |
| 目标岗位 | 初级前端 / 高级前端 / 架构师（影响亮点优先级） |

### 1.2 工作年限（⭐ 关键决策项）

**必须明确用户工作年限**，不同阶段的简历策略完全不同：

#### 🟢 1-3 年（初中级前端）
**定位**：证明"能独立完成任务，有工程意识，学习能力强"
- 亮点重心：落地过什么功能、解决过什么实际问题、用了哪些规范实践
- P2 条目同样重要，对这个段位有区分度
- 不要夸大：不要把"用了 Pinia"写成"设计了状态管理架构"
- 加分项：主动优化过某个问题、封装过被团队复用的组件、推动过某个规范

#### 🟡 3-5 年（高级前端）
**定位**：证明"能独立负责模块/系统，有技术判断力，能带动团队"
- 亮点重心：技术选型决策、系统性解决问题、对团队的影响
- 需要体现架构、设计模式、性能优化的量化成果
- 体现跨角色协作：和后端/产品/设计如何对齐

#### 🔴 5年+（资深/专家级）
**定位**：证明"能定义技术方向，降低团队整体复杂度，产生业务价值"
- 亮点重心：技术决策对业务的影响、工程体系建设、团队提效
- 必须有量化的业务成果（不只是技术指标）
- 体现前瞻性：为什么做这个技术选型，避免了哪些未来风险

> 根据用户年限，后续所有亮点的优先级判断、简历描述措辞、面试题深度均自动调整。

## Step 2 · AI 使用经历采集

⛔ **无论用户是否主动提及，都必须询问以下问题。**

AI 已成为前端岗位的重要加分维度：

**① 日常开发中用了哪些 AI 工具？**
- Cursor / GitHub Copilot / 通义灵码 / Windsurf / 其他
- 主要用于：代码补全 / 生成测试 / 写注释文档 / 改 bug / code review

**② 有没有在项目里集成 AI 能力？**
- 接入了 LLM API（OpenAI / Claude / 文心 / 通义等）
- 实现了 AI 对话框 / 流式输出（SSE / stream）
- 做过 AI 生成内容的前端渲染（Markdown 流式渲染、代码高亮）
- 做过 Prompt 管理页面 / AI 配置后台

**③ 有没有用 AI 辅助完成过某个具体任务？**
- 用 AI 生成了某个复杂组件的初始代码，再人工调优
- 用 AI 生成了单元测试用例
- 用 AI 辅助做了代码重构/性能分析

**④ GitHub 上有没有和 AI 相关的项目？**
- 独立做过 AI 工具/应用（提供 GitHub 链接）
- 开源项目有 Star / Fork
- 写过 AI 相关的技术博客/文章

> 有以上任一项，即在 Step 7 中生成 AI 相关简历描述。

## Step 5 · 八维度深度分析

逐一扫描，识别技术亮点与设计决策。**每个维度只输出存在的亮点，无亮点则跳过。**

### 5.1 组件设计
- 组件拆分粒度与职责划分（是否单一职责）
- 通信模式：props/emit / provide-inject / 状态管理 / 事件总线
- 复用策略：slot / HOC / render props / composables / hooks
- 设计模式：策略/观察者/工厂/代理/组合模式
- 组件文档与 TypeScript 类型定义质量

### 5.2 请求与数据层
- 请求封装：axios 实例 / 拦截器 / 统一错误处理
- 接口管理：模块化 / 自动生成（OpenAPI 等）/ 版本管理
- 数据缓存：SWR / stale-while-revalidate / 本地缓存
- 请求优化：取消重复请求 / 并发控制 / 请求队列
- 实时通信：WebSocket / SSE

### 5.3 状态管理
- 方案选型及理由
- 全局 vs 局部状态划分策略
- 持久化方案：localStorage / sessionStorage / pinia-plugin
- 异步状态模式：loading / error / data 三态处理

### 5.4 CSS 与样式工程
- CSS 方案及隔离策略
- 主题系统：CSS 变量 / Design Token / 暗色模式
- 响应式：媒体查询 / container query / 自适应布局

### 5.5 技术亮点（按场景识别）

| 场景 | 识别关键词 |
|------|-----------|
| 文件上传 | 分片/断点续传/拖拽/进度 |
| 权限系统 | 路由守卫/按钮权限/动态菜单/RBAC |
| 复杂交互 | 拖拽排序/在线编辑/富文本/可视化配置 |
| 数据可视化 | ECharts/D3/AntV/大屏适配 |
| 国际化 | i18n/语言包/动态切换 |
| 微前端 | qiankun/Module Federation/子应用通信 |
| 低代码/可视化搭建 | schema 渲染/拖拽布局 |

### 5.6 性能优化
- 加载优化：懒加载 / 预加载 / 骨架屏 / SSR / CDN
- 渲染优化：虚拟列表 / memo / shouldUpdate / 防抖节流
- 缓存策略：keep-alive / HTTP 缓存 / Service Worker
- 构建优化：代码分割 / Tree Shaking / 压缩 / 分包策略
- 网络优化：图片懒加载 / WebP / 资源预加载 / DNS 预解析

### 5.7 工程化与质量
- 构建配置深度（Vite/Webpack 自定义配置、插件开发）
- 代码规范：ESLint / Prettier / Husky / commitlint
- 测试覆盖：单元/组件/E2E 及测试策略
- CI/CD：自动化构建 / 部署 / 灰度发布
- Monorepo 管理（Turborepo / Nx 等）

### 5.8 错误处理与监控
- 错误边界（Error Boundary / errorCaptured）
- 全局异常捕获：拦截器 / unhandledrejection
- 日志与监控：Sentry / 自建上报
- 降级策略：接口超时 / 服务不可用兜底方案

## Step 6 · 隐性亮点挖掘

> **核心原则**：亮点不等于"难"。用户每天在写的代码，对面试官来说可能是加分项，但用户自己觉得"没什么好说的"。本步骤专门识别这类被低估的价值点。

### 6.1 六类隐性亮点识别（⛔ 逐类主动扫描，发现即标记）

#### A 类 · 「我只是封装了一下」
用户觉得只是"简单封装"，实则体现了抽象能力：
- 封装了带有错误处理 + loading 状态的请求 hook（哪怕只有 30 行）
- 把重复的表单验证逻辑抽成 composable/hook
- 封装了统一的弹窗/Toast 调用方式（不用每次 import）
- 对第三方组件做了二次封装，统一了默认行为

**写法**：强调"统一了 XX 行为，减少了 XX 重复代码，所有新页面开箱即用"

#### B 类 · 「这是产品要求的」
用户觉得只是在实现需求，实则包含技术决策：
- 实现了复杂的表单联动（多个字段相互影响）
- 实现了带权限的动态菜单（哪怕只用了 v-if）
- 实现了多标签页缓存（keep-alive + 路由配置）
- 实现了页面离开前的"未保存提醒"（beforeunload / 路由守卫）
- 搜索条件持久化到 URL query 参数

**写法**：描述"业务背景 + 你的实现思路"，体现对需求的理解和转化能力

#### C 类 · 「大家都这么做」
用户觉得是行业标配，实则是加分项（对 1-3 年候选人尤为重要）：
- 用 axios 拦截器统一处理 token 刷新
- 路由懒加载（import()）减少首屏包体积
- 用 env 文件管理多环境配置
- 用 composable/hook 分离业务逻辑和 UI
- 配置了 ESLint + Prettier + pre-commit hook

**写法**：写出"为什么这么做"而不仅是"做了什么"，体现工程意识

#### D 类 · 「踩过坑才做的」⭐ 最有说服力
用户解决过实际问题，但觉得"说出来显得自己之前很菜"：
- 发现接口重复调用，加了防重复提交
- 页面切换时发现内存泄漏，加了 onUnmounted 清理
- 时区问题导致日期显示错误，统一了日期处理工具函数
- 大量 if/else 难以维护，重构成了策略模式/配置化

**写法**："发现问题 → 分析原因 → 解决方案 → 防止复现"是最有说服力的简历素材

#### E 类 · 「只是提了个建议」
用户对团队有贡献，但觉得"不是自己独立完成的，不好意思写"：
- 推动引入了某个工具/规范（哪怕只是提议）
- 写了某个公共组件被团队复用
- 发现并修复了一个影响所有用户的 bug
- 整理了某模块的文档或注释

**写法**：用"参与/推动/协助"而非"独立主导"，诚实描述贡献范围

#### F 类 · 「产品说要能自己改」⭐ 重点挖掘
这是最容易被低估的一类。产品说"运营要能配置"，前端不得不把写死的东西做成可配置系统。

识别关键词：后台配置、运营配置、动态、灵活、不用改代码、开关、A/B、多套、皮肤、模板……

| 场景 | 识别特征 |
|------|---------|
| 内容/展示动态配置 | CMS 对接、富文本编辑、广告位配置、首页模块拖拽 |
| 功能开关与 A/B | Feature Flag、灰度发布、白名单机制 |
| 表单/流程动态配置 | JSON Schema 驱动表单、审批流配置、字段联动规则 |
| 主题/品牌定制 | CSS 变量、Design Token、多租户、暗色模式 |
| 导航/菜单动态化 | 后台驱动菜单、动态路由 addRoute |
| 消息/弹窗精细化 | 弹窗优先级队列、消息中心、可视化搭建 |
| 数据报表配置 | 自定义列、看板卡片拖拽、图表类型切换 |
| 价格/规则动态化 | 优惠券规则、SKU 选择器、限时活动配置 |

**F 类写法示例**：
```
✅ "设计并实现 JSON Schema 驱动的动态表单引擎，支持 20+ 种字段类型
和条件联动规则，产品可通过后台自助配置表单，新增业务场景无需前端
介入，需求交付周期从 3 天缩短至当天上线。"
```

### 6.2 主动追问（⛔ 分析完代码后必须提问）

向用户追问以下问题，挖掘代码之外的隐性价值：

1. "这个项目里有没有你觉得自己解决得比较好的问题，哪怕只是一个小功能？"
2. "有没有遇到过让你印象深刻的 bug 或者需求，花了比较多时间？"
3. "有没有你写的代码/组件，被同事用到了或者夸过好用？"
4. "有没有你当时觉得这样写'更优雅'，但也说不清楚为什么的地方？"
5. "有没有产品或运营提过'这个能不能不改代码就能改'这类需求？你是怎么实现的？"（→ F 类）
6. "有没有某个功能，产品说'不同用户/客户/场景要展示不一样的内容或流程'？"（→ F 类）
7. "有没有做过什么东西，上线之后产品/运营夸过'以后我们自己就能改了'？"（→ F 类）

# Phase 3 · 输出生成

## Step 7 · 简历描述生成

### 7.1 优先级规则

| 优先级 | 标识 | 判定条件 | 主要来源 |
|--------|------|---------|---------|
| P0 | 🔴 必写 | 体现架构能力、解决复杂问题、有量化成果 | 八维度分析 |
| P1 | 🟡 建议写 | 体现工程化思维、技术深度、团队协作；或 D 类隐性亮点 | 八维度 + 隐性挖掘 |
| P2 | 🟢 可选 | 常规技术实现；或 A/B/C/E/F 类隐性亮点 | 隐性挖掘为主 |

> 1-3 年候选人：P2 条目同等重要，不要忽略。

### 7.2 标准输出格式

```
🔴/🟡/🟢 [亮点标题] · 来源：[八维度/A-F类]
简历描述：[动词开头] + [技术方案] + [解决的问题] + [量化/定性成果]
```

**示例**：
```
🔴 权限系统设计 · 来源：八维度 5.5
简历描述：主导设计基于 RBAC 模型的前端权限系统，通过路由守卫 +
指令级按钮权限实现细粒度控制，支持动态菜单渲染，权限变更无需
重新部署，相比旧方案维护成本降低 60%。
```

### 7.3 AI 经历描述模块（根据 Step 2 采集结果生成）

**场景一：日常用 AI 工具提效**
```
"熟练使用 Cursor / GitHub Copilot 辅助开发，将其融入日常编码、
重构、单测生成等工作流，日均节省约 1-2 小时重复性工作。"
```

**场景二：项目中集成 AI 功能**
```
"基于 OpenAI API 实现智能客服对话模块，使用 SSE 实现流式输出，
支持 Markdown 实时渲染和代码高亮，响应延迟感知降低 60%。"
```

**场景三：GitHub 有 AI 项目**
```
"独立开发并开源 XX 工具（GitHub X stars），基于 XX 模型实现了
XX 功能，在社区获得广泛关注。"
```

### 7.4 转型补充模块（根据 Step 3 采集结果生成）

**转向全栈**：
```
"具备全栈开发能力，独立负责过 XX 项目的前后端完整开发，
后端使用 Node.js + Express，数据库使用 MySQL，
独立完成从需求到上线的全流程。"
```

**转向后端**：
```
"具备完整的 Web 全栈视角，深度参与前后端联调，
熟悉 RESTful 接口设计规范，能快速理解后端数据结构
和业务逻辑，上手新后端项目成本低。"
```

**转向产品**：
```
"具备前端工程师背景，深度理解产品技术实现路径，
参与过多个需求从 PRD 到上线的完整周期，
多次在评审阶段识别技术风险并提出替代方案。"
```

### 7.5 无量化数据时的替代写法

| 场景 | 替代表述 |
|------|---------|
| 无性能数据 | "解决了用户反馈的 XX 问题" / "消除了 XX 场景下的卡顿" |
| 无效率数据 | "该组件被团队 X 个项目复用" / "减少了每个新页面约 XX 行重复代码" |
| 无用户规模 | "支撑了核心业务流程" / "覆盖了 XX 模块的所有用户" |
| 无对比数据 | "统一了团队的 XX 实践，降低了新成员上手成本" |

## Step 8 · 面试题生成

针对 P0 和 P1 亮点，每个亮点生成以下四层面试题：

```
【结论先行】

【技术原理/方案选型】
- 核心机制是什么

- 关键实现细节

【项目中的实际应用】
- 具体场景：（描述业务背景）
- 技术实现：（描述你的方案）
- 遇到的挑战：（描述难点）
- 解决方式：（描述解法）

【结果与收益】
- 量化：（性能/效率/稳定性指标）
- 定性：（如无量化数据，描述业务价值）

【延伸/反思】（可选，用于加分）
- 方案的局限性是什么
- 如果重来，会有哪些改进
```

**完整示例（虚拟列表）**：

```
【结论先行】
用虚拟列表解决了万级数据表格的渲染卡顿问题，首屏渲染从 3.2s 降至 0.8s。

【技术原理】
虚拟列表只渲染可视区域内的 DOM 节点，通过动态计算 scrollTop
偏移量来决定渲染哪些条目。对比了 vue-virtual-scroller 和自研方案，
最终选择自研，因为需要支持动态行高，第三方库支持不够好。

【项目实际应用】
场景：审批记录列表，单次查询返回 5000+ 条数据，产品不允许分页。
实现：用 IntersectionObserver + 高度预估 + 动态修正三步走。
挑战：动态行高导致滚动位置跳动。
解法：用 ResizeObserver 实时修正已渲染行的高度缓存。

【结果】
首屏渲染 3.2s → 0.8s，内存占用减少约 70%，用户投诉归零。

【反思】
若重来，会优先考虑后端分页 + 无限滚动方案，前端虚拟列表维护成本偏高。
```
